# KNOWN_ISSUES.md

## Риски CI

- **SDK-зависимость Liquid Glass**: `glassEffect`, `GlassEffectContainer` и
  `.buttonStyle(.glass/.glassProminent)` объявлены в SDK iOS 26. Все вызовы
  обёрнуты в `#available(iOS 26.0, *)` с fallback на `Material`
  (`Luma/DesignSystem/LumaGlass.swift`), но **сам код должен компилироваться**,
  а значит Xcode на раннере GitHub Actions обязан включать iOS 26 SDK. Workflow
  `build.yml` выбирает самый новый установленный Xcode (`sort -V | tail -1`) —
  если на раннере окажется только более старый Xcode без iOS 26 SDK, сборка
  упадёт на строках с `.glassEffect`/`.glass`. В этом случае нужно либо
  дождаться обновления образа `macos-15`, либо временно закрепить более новый
  `macos-*` раннер в workflow.
- **Имя симулятора для тестов**: `-destination` в `build.yml`/`tests.yml`
  указывает `name=iPhone 17` (актуально на момент написания — на раннере с
  Xcode 26.3 симулятор `iPhone 15` уже отсутствует, доступны только iPhone 16/
  16e/17/Air/SE). Имя симулятора должно существовать в образе раннера; если
  Apple в очередной раз обновит линейку, сборка упадёт на этапе выбора
  destination с списком доступных `{ platform:iOS Simulator, name:... }` в
  логе — тогда нужно взять из этого списка актуальное имя.
- **Валидация Swift-макросов**: с добавлением `LocalLLMClient` (см. переход
  к реальному инференсу) в граф зависимостей попадает пакет со Swift-макросом
  (`LocalLLMClientMacrosPlugin`). Xcode 15.3+ требует явно «доверять» плагинам
  макросов из внешних пакетов — интерактивно это делается кликом в UI, а на
  headless CI сборка падает с `error: Macro "..." must be enabled before it
  can be used`. Все вызовы `xcodebuild build/test/archive` во всех четырёх
  workflow-файлах передают флаг `-skipMacroValidation`, который снимает это
  требование для CI. Если в будущем добавится ещё один пакет с макросами —
  флаг уже общий, менять ничего не нужно.

## Установка unsigned IPA

`build.yml` и `release-unsigned.yml` собирают **неподписанный** IPA
(`CODE_SIGNING_ALLOWED=NO`). Обычный iPhone не установит такой файл напрямую —
нужен пересборщик подписи на стороне пользователя (например AltStore,
Sideloadly или аналог с собственным Apple ID). Это ограничение процесса
сборки без Mac и без Apple Developer Program, а не баг Luma.

## Дизайн / UI

- `ModelDetailView` использует принудительное разворачивание `model!` в
  нескольких computed properties, полагаясь на то, что экран не рендерится
  при `model == nil` (заглушка `ContentUnavailableView` в `body`). Это
  приемлемо для Этапа 1 на моковых данных, но при переходе на реальный
  каталог моделей (Этап 5) стоит заменить на безопасный `guard let`.
- `HistoryView` сортирует по закреплению нестрогим компаратором
  (`{ $0.isPinned && !$1.isPinned }`), что не является строгим слабым
  порядком в строгом смысле, но при небольшом количестве диалогов на моках
  ведёт себя предсказуемо. Пересмотреть при реальных данных большого объёма.

## Реальный инференс (внедрён раньше графика Этапа 5)

- **Загрузка моделей «резюмируется» только на уровне целого файла.**
  `ModelDownloader` пропускает файлы, уже существующие на диске от прошлой
  прерванной попытки, но не сохраняет частичный прогресс внутри
  недокачанного файла (`URLSessionDownloadTask` создаётся заново на каждый
  файл, без `resumeData`). Для файла весов (~0.9 ГБ) при обрыве соединения
  придётся перекачать его целиком, а не докачать остаток. Настоящий
  побайтовый resume — доработка на будущее, не выдаваемая за уже
  реализованную.
- **Нет проверки свободного места перед загрузкой.** `ModelDownloader`
  создаёт директорию и начинает качать без предварительной проверки
  доступного места на диске — при нехватке места система прервёт запись
  файла с обычной ошибкой `NSFileWriteOutOfSpaceError`, но заранее об этом
  не предупреждает.
- **Оценки качества моделей** (`russianQuality`, `codeQuality` и т.д. в
  `Luma/Models/LocalModel.swift`) — качественные, основанные на общеизвестных
  характеристиках публикаций Qwen2.5, не измерены на устройстве самим
  Luma (у приложения ещё нет своих evals).
- **`DiagnosticsSnapshot`** (`Luma/Models/PerformanceProfile.swift`) остаётся
  моковым — он питает только служебный экран «Диагностика» (контекстное
  окно, скорость токенов и т.п., для которых ещё не подключён реальный
  агент), а не пользовательские ответы. Реальные данные устройства для
  ответов агента идут через отдельный `DeviceStatusProvider`.
